# Личное окружение AI-агентов для разработки и исследований

- Статус: active
- Дата: 2026-07-19
- Тема: простое и эффективное взаимодействие пользователя с AI-агентами для разработки и исследований
- Выборка: 55 уникальных результатов, 47 кандидатов после дедупликации, 6 детальных карточек

## Цель

Определить воспроизводимую структуру личного окружения, в которой пользователь формулирует цель, агент исследует и выполняет работу, а результаты, ограничения и следующий шаг остаются понятными между сессиями.

- Запросы: `настройка личного окружения AI-агентов разработка исследования`, `как эффективно работать с AI-агентами Codex Claude Code`, `personal AI agent development research workspace`, `AI coding agent environment setup best practices`.
- Режим отбора: `auto`; найдено 55 уникальных результатов, выбран предел 50, в итоговую выборку попало 47 роликов.
- Глубокий разбор: 6 приоритетных карточек по метаданным, описаниям и доступности публичных субтитров.
- Ограничение: YouTube Data API не отдаёт текст субтитров. У двух роликов API пометил субтитры как доступные, но публичная страница YouTube фактически показала `Субтитры недоступны`; содержательные выводы из них не делались. Это не означает, что текст недоступен в YouTube Summary: добавлен локальный захват текста из уже отображённого виджета.

## Подтверждённые выводы

1. **Постоянный контекст и повторяемые действия нужно разделять.** Claude Code документирует `CLAUDE.md` как контекст на каждую сессию, а skills как вызываемые повторяемые знания и workflows. Это подтверждает модель: короткие общие правила отдельно, объёмные процессы отдельно. Источники: [Claude Code overview](https://code.claude.com/docs/en/overview), [Extend Claude Code](https://code.claude.com/docs/en/features-overview).
2. **Специализацию стоит включать по необходимости, а не превращать всё в постоянный промпт.** Официальная документация Claude Code различает skills, MCP и subagents: skills подходят для повторяемой процедуры, MCP для внешнего источника или действия, subagent для изолированной узкой работы. Источник: [Extend Claude Code](https://code.claude.com/docs/en/features-overview).
3. **Переносимость среды разработки достижима через Dev Container.** Development Container Specification определяет контейнер с настройками, специфичными для разработки. Это подходит для воспроизводимых рантаймов и инструментов, но не заменяет правила агента или базу знаний. Источник: [containers.dev](https://containers.dev/).
4. **Карточка задачи и отчёт исследования уменьшают потерю контекста между сессиями.** Это архитектурное решение данного репозитория, а не утверждение из роликов: `work/active/` хранит цель, состояние и следующий шаг; `knowledge/` хранит доказанные выводы.

## Не подтверждено

- Конкретные техники и утверждения шести роликов: полные тексты и проверенные таймкоды не получены.
- Заявления авторов о лучшей модели, автономности, стоимости, лимитах, готовых сборках и партнёрских продуктах.
- Возможность массово получать новые расшифровки через Glasp MCP: в этой сессии его инструмент не был доступен; официальный read-only контракт библиотеки не доказывает пакетное получение новых транскриптов.

## Практические рекомендации

1. **Одна цель на сообщение.** Пользователь задаёт цель, критерий готовности и ограничения; агент сам собирает рабочий план и возвращает результат с доказательствами.
2. **Три уровня памяти.** Глобальные правила и безопасность — в `AGENTS.md`; правила конкретного проекта — в файле проекта; повторяемые процедуры — в skills. Не дублировать объёмные инструкции во всех трёх местах.
3. **Внешние инструменты только по задаче.** MCP подключается для конкретного доступа: поиск YouTube, GitHub, документация. Оставлять read-only режим по умолчанию и не включать лишние серверы постоянно.
4. **Явный рабочий след.** Для длинной работы создавать `work/active/<topic>.md`; в конце сохранять короткий отчёт в `knowledge/`. Это позволяет продолжить работу без повторного обучения агента.
5. **Сначала воспроизводимость, потом автономность.** Проекты с одинаковыми зависимостями и командами можно оформить в Dev Container; автоматизацию и subagents добавлять лишь после того, как базовый цикл «цель -> изменения -> тесты -> отчёт» стабилен.

## Стоимость и ограничения

- **YouTube Data API:** текущая официальная документация указывает отдельный дневной лимит 100 вызовов `search.list` и общий лимит 10 000 единиц для остальных методов. Один `search.list` стоит одну единицу в своём лимите. В этом исследовании выполнено четыре поисковых вызова и один запрос деталей. Источники: [Quota Calculator](https://developers.google.com/youtube/v3/determine_quota_cost), [search.list](https://developers.google.com/youtube/v3/docs/search/list).
- **YouTube Summary:** может быть источником текста, если расширение фактически отображает расшифровку, в том числе при отсутствии публичной панели субтитров YouTube. Локальный `YouTube Research Capture` получает текст из уже отображённого виджета и обрабатывает очередь до шести роликов в фоне. При ошибке или лимите сервиса очередь помечает ролик как недоступный и не пытается обойти ограничение. Не утверждаю, что бесплатный тариф покрывает массовое извлечение: бесплатные лимиты и доступные функции должны быть сверены по официальному тарифу до использования.
- **Бесплатная альтернатива:** текущий локальный YouTube MCP получает публичные метаданные, описания и ссылки в пределах квоты API. Он не получает закрытые или недоступные субтитры.

## Следующее действие

Однократно загрузить `YouTube Research Capture` в Chrome и перезапустить Codex. Затем повторный запуск `topic-research` сам создаст очередь из шести роликов, получит доступный текст из YouTube Summary, обновит карточки короткими цитатами с таймкодами и проверит технические утверждения по первичной документации.
