## Практическая схема личной среды

1. **Глобальный слой.** Короткий `~/.codex/AGENTS.md`: язык, стоимость, безопасность, различение фактов и предположений, формат отчёта. Локальный `~/.codex/config.toml`: включённые MCP и только имена секретов; секреты лежат в переменных окружения.
2. **Слой проекта.** `AGENTS.md` в репозитории: команда запуска, тесты, архитектурные ограничения и критерии готовности. Файлы проекта важнее глобальных предпочтений.
3. **Слой повторяемых процессов.** Skills для исследования, проверки кода, подключения MCP и ведения карточек. Они снимают с пользователя необходимость заново объяснять последовательность действий.
4. **Слой памяти работы.** `work/active/` хранит цель, решения, ссылки и следующий шаг; `knowledge/` хранит только краткие доказательства и выводы. Полные расшифровки и секреты не коммитятся.
5. **Слой выполнения.** Для большой или неоднозначной работы: сначала план и критерий готовности, затем выполнение; отдельно -- исследование источников, отдельно -- реализация, отдельно -- тесты/review.
6. **Слой автоматизации.** Автоматизировать только процесс, который уже понятен и проверен вручную. Каждая автоматизация должна иметь вход, ожидаемый результат, лимит стоимости, лог ошибок и безопасный останов.

## Не подтверждено

- Утверждения авторов о «лучших» моделях, конкретных ценах, экономии времени, качестве сторонних MCP и их безопасности. Они не проверялись по документации поставщиков.
- Полезность TestSprite, Context7, n8n и прочих упомянутых сервисов именно для этой среды. Они являются кандидатами, а не установленными инструментами.
- Рекомендация вести десятки специализированных агентов из `0SFV9KOP9Ss`. Для личной среды это может создать больше конфигурационной нагрузки, чем пользы.
- Короткий ролик `q92jEz-Y7Lo`: полученный текст не содержит достаточного материала для вывода.

## Практические рекомендации

1. Для любой заметной задачи писать одну фразу цели, релевантный контекст, ограничения и проверяемый критерий `готово когда`.
2. Не смешивать новую задачу с длинной историей предыдущей: завершать работу коротким отчётом и начинать независимый этап с карточки задачи и нужных ссылок.
3. Держать общие правила короткими; процесс исследования, тестирования или подключения сервиса вызывать отдельным skill, а не вставлять его целиком в глобальные инструкции.
4. Подключать MCP только для конкретной потребности и включать стоимость/лимит в отчёт до массового запуска.
5. Перед автоматизацией описывать ручной процесс, ожидаемый результат, ошибку и точку остановки. После этого автоматизировать небольшой проверяемый фрагмент.
6. Считать задачу по изменению кода завершённой только после релевантных тестов, review diff и явной записи того, что проверить не удалось.

## Стоимость и ограничения

- **Supadata:** официальный сайт заявляет 100 бесплатных запросов в месяц без банковской карты; этот запуск израсходовал 11 запросов на извлечение шести источников и повторный анализ пяти из них. Batch endpoint у провайдера отмечен как платный, поэтому локальный MCP делает ограниченные последовательные запросы. [Supadata YouTube Transcript API](https://supadata.ai/youtube-transcript-api), [тарифы](https://supadata.ai/pricing)
- **Риск стороннего провайдера:** Supadata получает URL роликов и в своих условиях оставляет соблюдение правил YouTube и права на контент ответственностью пользователя. Полные тексты не сохраняются в репозиторий. [Условия Supadata](https://supadata.ai/terms)
- **Бесплатная альтернатива:** YouTube Data API для поиска/метаданных и открытые субтитры, но он не даёт текст субтитров произвольных роликов. Для собственных или авторизованных файлов можно использовать локальный Whisper, но это не заменяет легальное получение аудио чужих роликов.

## Следующее действие

Внедрены компактный шаблон больших задач и локальный бюджет Supadata: `80` запросов в месяц, из которых `15` уже учтены. После перезапуска Codex проверять бюджет через `youtube_transcript_budget` перед новым исследованием. Не добавлять новые MCP и не менять безопасность без отдельной проверки стоимости, прав доступа и назначения.
